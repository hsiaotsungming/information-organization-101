RAG：讓 LLM 根據資料回答

檢索增強生成（Retrieval-Augmented Generation, RAG）會先從外部知識庫或文件中找回與問題相關的資料，再讓大型語言模型根據這些資料生成答案，使回答更接近可查證、可追溯的內容。

1. RAG 是什麼 2. 找資料再回答 3. 圖書館情境 4. RAG 的限制 5. 小練習

本單元目標

  • 理解什麼是 RAG。
  • 說明 RAG 為什麼能提升 LLM 回答時的可查證性。

RAG 是什麼

RAG 可以理解為「先檢索，再生成」，也就是先找回相關資料，再根據資料產生較有依據、較容易追溯來源的回答。

大型語言模型作答時，主要根據訓練時學到的語言模式與當下提示進行生成。在 RAG 中，系統會先依據問題去檢索資料庫，再把找到的內容一併提供給大型語言模型來生成答案。因此，模型不是憑記憶或語言機率回答，而是根據檢索回來的資料來生成答案。

在 RAG 中，使用者提出問題後，系統會根據問題進行語意搜尋，找出意思相近、內容相關的資料。這個搜尋通常會使用向量資料庫（vector database），查詢原始文件或新資料經過切分與向量化後的表示。右下角代表內容會匯入、處理、切分與向量化，最後存入向量資料庫。接著，系統會將使用者問題與檢索到的內容組成 prompt，再送給 LLM。LLM 依據這些補充資料產生回答後，系統再經過 post processing 整理內容，最後將 response 回傳給使用者。整體來說，RAG 的重點是讓 LLM 不只依靠原本的訓練知識，而是能參考外部資料，產生較符合資料內容的回答。

RAG 的基本流程

我們可以把上面的 RAG 流程簡化成三步。第一步是由理解問題，辨識查詢中的重點條件，例如主題、人物、作品、時間或資料範圍。第二步是檢索，把和這些條件相關的資料從資料庫中找出來。第三步則是生成，讓大型語言模型根據這些被找回來的內容整理出回答。

1. 理解問題

模型或檢索器先判斷問題在問什麼，辨認關鍵條件與限制。

2. 找回資料

系統根據條件，從資料來源中找出最相關的內容片段或紀錄。

3. 整理回答

大型語言模型根據找回來的內容，用自然語言回覆使用者。

對使用者來說，這看起來像一次問答；但在系統背後，其實已經先做了一次檢索工作。也因為如此，RAG 的品質不只取決於模型，也取決於資料有沒有被好好整理、切分、描述與索引。

舉例來說，當使用者問「A公司今年的員工旅遊補助是多少？」時，大型語言模型可能只能根據訓練資料和語言模式推測。而 RAG 會先到資料庫中進行檢索，查找關於這個年份的員工旅遊補助資料。如果有找到適當資料，模型就會根據這段內容回答。

簡言之，RAG 在大型語言模型回答前，先找到與問題相關的文本、紀錄、條目或知識片段，讓大型語言模型用來生成回答。這樣的設計適合需要根據特定資料、館藏、文件內容來回答的情境。

先找資料，再讓模型回答

RAG 讓回答不是憑空生成，而是大型語言模型依據具體資料整理成回覆。換句話說，系統不是直接要求模型憑既有知識回答，而是先根據使用者的問題，包含文本、紀錄、條目、摘要、主題標示或評論，再把這些資料一併提供給模型來回答。

例如，使用者問「台積電最近一季的營收表現如何？」一般大型語言模型本身不一定掌握最新財報或即時公告，也不應該只根據過去訓練資料推測。RAG 系統可以先從公司財報、法說會簡報、重大訊息公告或財經資料庫中，檢索台積電最近一季的營收、毛利率、淨利、每股盈餘與管理層說明，再把這些內容提供給模型。模型接著根據檢索到的資料，整理出該季營收是成長或下滑、主要原因是什麼，以及公司對後續展望的說法。如果系統設計得更完整，也可以同時附上資料來源與公告時間，讓使用者知道這段回答是依據哪一份財報或哪一次法說會資料產生的。

這麼做可以讓系統的回答更容易連回資料來源，使用者不只會得到整理後的文字，也可以看到回答的依據，這使得回答可以被檢查、追溯與修正。當公司發布新的財報、公告或市場資料時，系統也可以重新檢索最新內容，讓回答更貼近目前可用的資料。

因此，RAG 適合應用在需要依據資料回答的情境。它的重點不是讓模型取代資料庫，而是讓模型在資料庫檢索結果的基礎上，並協助使用者理解、比較與整理資訊。

RAG 的重點

RAG 在模型生成回答前，先加入一層資料檢索與依據。它的作用是讓回答盡量建立在可查找、可追溯的資料之上，並讓使用者更容易檢查答案來自哪些文件、紀錄或資料來源。

圖書館中的 RAG

問題例：我想找有館內有那些內容有提到神啊鬼的中文輕小說。

RAG 可以基於館藏目錄中的書目紀錄和外部連結內容，作為資料庫來檢索並提供LLM做生成。當使用者提出自然語言問題時，系統先從這些資料中找出相關內容，再讓模型將結果整理成更容易閱讀的回答。這樣的設計能降低使用者操作系統的難度，也能協助讀者服務人員回應查詢需求，例如推薦相關館藏、引導讀者找到合適的資料庫、整理主題資料，或說明某類資源的使用方式。

由於會使用資料庫的內容做為生成，資料庫的內容會直接影響生成答案的品質。如果像是題名、作者、主題、版本、語言、館藏地與典藏狀態等資訊正確，系統就更容易找回正確內容。反之，如果資料庫的內容本身就不正確，那 RAG 能提供的幫助就會打折扣。

換句話說，RAG 並不是只靠模型能力就能運作良好，它同時仰賴資料庫中的資料品質。良好的資料品質是讓系統能夠順利運作的基礎。資料組織得越清楚，模型後續整理出來的回答也越有機會準確、穩定，並且能夠連回可靠的來源。

資料與 RAG

RAG 的品質不只取決於大型語言模型的能力，也取決於資料內容的品質。

RAG 的限制

從先前內容可知，雖然 RAG 能補強大型語言模型脫離資料作答的問題，但它不是萬靈丹。由於模型生成答案前，必須先依賴系統找回相關資料，因此如果檢索時找錯資料、漏掉關鍵資料，或找回來的內容本身不完整、不一致，最後生成的回答仍可能出現偏差。

此外，雖然 RAG 具有語意檢索能力，但對於實體關係的判斷上，仍然有不足之處－特別是難以由文字內容中直接呈現的語意或實體關係。由於 RAG 是以文字做為檢索基礎，在前句所說的情形中，可能無法提供足夠精準的關聯判斷。例如，系統可能找得到某位作者、某本書或某個主題的相關文字，但不一定能正確理解各相關作品之間的關聯與差異。若應用情境需要處理較複雜的實體關係，糋需要搭配進一步的結構化資料，像是詮釋資料、權威控制、知識圖譜等。

小練習

  1. 用自己的話說明 RAG 為什麼可以理解為「先找資料，再回答」。
  2. 想一個圖書館問答情境，列出系統在回答前應該先找哪些資料。
  3. 說明如果資料本身整理得不清楚，RAG 可能會受到哪些影響。
